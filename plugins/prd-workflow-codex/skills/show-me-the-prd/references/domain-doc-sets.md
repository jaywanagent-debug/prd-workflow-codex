# Domain Doc Sets — 4종 문서 동적 생성 가이드

> **개수 4는 사용자 예측가능성의 본질로 유지** (사용자가 항상 4종 받음).
> **4종이 어떤 4종인지는 매번 도메인을 보고 동적 결정**.
> 본 문서는 enum 카탈로그가 아니라 **동적 생성 절차 + 참고 예시**다.

---

## ⚠️ 본 문서 사용 원칙

- ❌ **메뉴처럼 사용 금지**: "B2B SaaS면 이 4종, 이커머스면 저 4종" 식 enum lock-in 금지
- ✅ **예시처럼 사용**: "이런 식으로 도메인별 변환 사례를 본 다음, 사용자 PRD 요청에 맞게 4종을 매번 정의"

---

## 핵심 원칙

1. **개수 4 + README = 5 파일 항상 출력** (사용자 예측가능성 본질)
2. **4종 정의는 매번 동적** — 사용자 도메인 분석 → 4종 즉석 정의
3. **항상 다음 4 차원이 모두 채워지도록**:
   - **요구사항** (무엇을 만들지)
   - **구조** (데이터/모델/시스템)
   - **도메인 특화** (이 도메인 특수 사항: 규제/흐름/단계)
   - **AI 행동 규칙** (개발 시 지켜야 할 규칙)
4. **본 문서의 예시는 메뉴가 아닌 패턴 학습용**

---

## 동적 생성 절차

Turn 1 인터뷰에서 도메인 추정 후:

### Step 1: 도메인 핵심 추출
- 누가 쓰는 제품? (사용자 / 머천트 / 의료진 등)
- 핵심 데이터는? (트랜잭션 / 콘텐츠 / 사용자 데이터)
- 도메인 특수 사항은? (결제 / 규제 / 학습 / 의료 등)
- 외부 제약은? (GDPR / HIPAA / 결제 표준 등)

### Step 2: 4 차원 매핑
| 차원 | 결정 사항 |
|---|---|
| 요구사항 | 항상 PRD 1종 (보편) |
| 구조 | DATA_MODEL / PIPELINE / CURRICULUM / 등 — 도메인 핵심 구조 |
| 도메인 특화 | PHASES / COMPLIANCE / PAYMENT_FLOW / TENANCY / LICENSE 등 |
| AI 행동 규칙 | 항상 PROJECT_SPEC 1종 (보편 — AI 협업 가드) |

### Step 3: 도메인 특화 명명

각 차원의 슬롯을 도메인에 맞게 명명. 아래 §참고 예시 패턴 학습.

### Step 4: 사용자 confirm (`shared/questioning-policy.md §A` — 채팅 번호 블록)

```
이 PRD는 [추정 도메인] 같아 보입니다. 다음 4종 문서를 생성할까요?
- 01_PRD.md (요구사항)
- 02_[X].md (구조 — 도메인에 맞게 결정)
- 03_[Y].md (도메인 특화 — 결제/규제/학습 흐름 등)
- 04_PROJECT_SPEC.md (AI 행동 규칙)
+ README.md (네비게이션)

1. 이대로 진행
2. 다른 4종으로 변경 (문장으로 적어주세요)
(모르면 1번으로 진행할게요)
```

## 참고 예시 — 동적 생성 사례 모음

> **이 예시들은 메뉴가 아니다.** 같은 도메인이라도 사용자 아이디어 세부에 따라 4종이 다르다.
> "이런 식으로 4 차원이 도메인 슬롯에 매핑되는 사례를 본다"는 학습용 샘플.

| 03 | PHASES.md | 도메인 특화 (단계 분리) |
| 04 | PROJECT_SPEC.md | AI 행동 규칙 |

## 도메인 추정이 모호한 경우

```
이 도메인은 추정 어렵네요. 4 차원의 슬롯을 함께 채워볼까요?
(개수 4는 예측가능성의 본질이라 유지):

01. PRD.md (요구사항 — 보편)
02. ____.md (구조: 어떤 데이터/모델/시스템?)
03. ____.md (도메인 특화: 이 프로젝트만의 특수 사항?)
04. PROJECT_SPEC.md (AI 행동 규칙 — 보편)

사용자가 답변 못 하면 → 예시 0 (기본 set) 사용
```

## ⚠️ 절대 규칙

1. **항상 4종 + README = 5 파일 출력** (사용자 예측가능성)
2. **4 차원이 모두 채워져야 함** (요구사항 / 구조 / 도메인 특화 / AI 행동 규칙)
3. **본 문서를 메뉴로 사용 금지** — 예시는 패턴 학습용
4. **도메인 추정이 모호하면 예시 0 (기본 set) 사용**
5. **사용자 confirm 후 4종 정의 확정** — silent execution 금지
6. **새 도메인 사례를 본 문서에 추가하지 말 것** — 이 문서는 enum 카탈로그가 아닌 패턴 예시집

---

## 본 문서가 fossil 되지 않는 가드

- 본 문서의 예시 8개는 **상한이 아니라 학습용 샘플**
- 카탈로그에 "있어야" 그 4종 만드는 게 아니라, 매번 동적 정의
- 사용자가 새 도메인 가져오면 → 본 문서를 늘리지 말고, 그 자리에서 4종 정의
